# Issue Report Diagnostics

This document defines the diagnostic bundle generated by the in-app
Report Issue flow.

The goal is that a public bug report gives maintainers enough runtime
context to start fixing the issue without first asking the reporter to
reproduce it, while still avoiding accidental disclosure of secrets or
local machine details.

## Problem Class

Issue reporting is a mature diagnostic bundle problem:

- Products such as VS Code collect app, environment, extension/runtime,
  and log context in their issue reporting paths.
- GitHub issue forms provide structured user input, but the application
  must still generate the technical context.
- Error and crash reporting systems scrub sensitive fields before data
  leaves the local boundary.

OpenCove should therefore treat issue reports as a structured diagnostic
bundle, not as a long free-form string.

## Owners And Boundaries

| State or action | Owner | Notes |
| --- | --- | --- |
| User intent | Renderer issue report dialog | Issue kind, summary, details, local path opt-in. |
| Diagnostic collection | Main issue report infrastructure | Reads logs, app state, worker state, process snapshots. |
| Section schema and report formatting | Issue report application layer | Defines section shape, budgets, markdown output, GitHub output. |
| Sanitization | Issue report application layer | Final mandatory pass before writing files, copying, or opening GitHub. |
| Report file storage | Main issue report infrastructure | Writes under app-owned `issue-reports/`. |

Renderer must not read diagnostic files directly. Main collects facts and
runtime observations; the application layer turns them into a bounded,
redacted report.

## Diagnostic Sections

Issue reports use a section registry. Each section is independently
collectable and independently allowed to fail:

```ts
type IssueReportDiagnosticSection = {
  id: string
  title: string
  sensitivity: 'safe' | 'redacted' | 'local-paths-optional'
  github: 'summary' | 'excerpt' | 'omit'
  status: 'available' | 'unavailable'
}
```

The standard bundle should include:

- `report_meta`: report id, created time, kind, budget and sanitizer version.
- `user_context`: user summary/details, active workspace/space names, and
  local paths only when explicitly opted in.
- `app_runtime`: OpenCove, Electron, Chrome, Node, platform, arch, pid,
  packaging, locale/timezone when available.
- `update_state`: app update state.
- `worker_state`: home worker mode, web UI status, endpoint availability,
  hostname/port, with tokens always hidden.
- `agent_state`: default provider/model, full access flag, terminal profile,
  provider availability, executable override presence without executable paths.
- `process_snapshot`: process tree status, process summary, key process rows,
  and collector notes.
- `log_bundle`: recent tails of runtime, terminal, and PTY host logs.
- `diagnostics_manifest`: section availability, truncation, byte counts,
  and collection errors.

## Invariants

1. A failed section must not fail report generation. It must render as
   unavailable with a concise error.
2. Every output path must run the final sanitizer: report file, returned
   markdown, clipboard text, and GitHub issue body.
3. Local paths are hidden by default. The local path opt-in only affects
   paths; credentials and tokens are always redacted.
4. Logs must include metadata: missing, empty, read failed, original bytes,
   included bytes, and truncation status.
5. GitHub issue body must include a usable diagnostic summary and log
   excerpts. It must not only ask the reporter to paste a local file.

## Sanitization Policy

The sanitizer is a final safety pass. Collectors should avoid collecting
secrets where possible, but they must not rely on that alone.

Always redact:

- `Authorization`, `Bearer`, `Basic`, `Cookie`, and `Set-Cookie` values.
- Env, JSON, and query values with secret-like keys:
  `TOKEN`, `SECRET`, `PASSWORD`, `PASS`, `API_KEY`, `ACCESS_KEY`,
  `PRIVATE_KEY`, `SESSION`, `JWT`, `AUTH`.
- URL credentials such as `https://user:password@example.com`.
- Common long token shapes such as `sk-*`, `ghp_*`, `github_pat_*`, and JWTs.
- Private key blocks.
- Worker and endpoint tokens, regardless of local path opt-in.
- Known local paths unless `includeLocalPaths` is enabled.

Known local paths should include at least:

- user home directory,
- app `userData`,
- active workspace path,
- active space path.

## Budgets

The report has two budget tiers:

- Full local report: large enough for real diagnosis, but bounded so copying
  and file viewing remain practical.
- GitHub issue body: compact enough for URL prefill, but still useful without
  asking for another reproduction.

Each log excerpt must record the original byte count, included byte count,
and whether the content is a tail excerpt.

Budget changes are allowed, but must update tests that assert truncation and
must preserve the invariants above.

## Extension Rules

When adding a new diagnostic source:

1. Add a new section collector instead of appending ad hoc strings to the
   report builder.
2. Decide whether the section is `safe`, `redacted`, or
   `local-paths-optional`.
3. Decide whether GitHub gets `summary`, `excerpt`, or `omit`.
4. Add unavailable/error behavior and a manifest entry.
5. Add the lowest meaningful test:
   unit for formatting/sanitization, contract for IPC shape, integration for
   cross-boundary collection, or E2E for user-visible issue reporting.

Do not add raw database dumps, raw topology files, browser history, full
terminal transcripts, or unbounded command lines to the standard bundle.
Add summarized, redacted, bounded sections instead.
